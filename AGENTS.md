## Main Goal

The current goal is to generate a good Bridgeland Verso manual from a
mechanically-generated shadow copy:

1. compute the trusted formalization base directly from the compiled
   environment,
2. copy the Bridgeland repo into a shadow repo,
3. annotate the selected declarations with `-- ANCHOR:` markers,
4. patch the shadow repo to add `verso` and comparator-manual targets,
5. render a usable Verso manual site with good SubVerso highlighting,
6. keep the TFB declaration list filtered and ordered sensibly,
7. drop theorem proofs in the rendered TFB blocks without mangling the Lean,
8. avoid expensive cold rebuilds on every shadow regeneration.

## What Has Been Tried

### Attempt 1: plain fenced code blocks

This produced pages, but without the Verso / SubVerso code experience. It was
not good enough.

### Attempt 2: anchored external `module` blocks in a shadow repo

This was the right overall direction, but it initially had several bugs:

- copied files were hard-linked instead of copied,
- heading nesting was invalid,
- some signatures were truncated,
- duplicate section tags showed up for names like `obj₁`, `obj₂`, `obj₃`,
- the TFB slice included too many autogenerated declarations.

These were fixed in `Site.lean`.

### Attempt 3: default `defSite` for TFB blocks

This caused bad local example-definition linking in the TFB HTML, with many
`title="Definition of example"` artifacts. The current TFB path disables
`defSite`.

### Attempt 4: HTML post-processing to strip proofs

This was fragile and should not be revived. It broke on `where`-style theorem
values and destroyed the rich Lean highlighting.

### Attempt 5: rewrite copied shadow source to `:= by sorry`

This was also rejected. It made the copied source diverge from the anchored
source text and created mismatch problems.

### Attempt 6: custom `prooflessModule`

This is still the right design direction. It works at render time against the
real anchored highlighted block instead of rewriting source or post-processing
HTML.

### Attempt 7: `signature` blocks for theorem heads

This was tried after the user asked for a less fragile proof-dropping path.
It failed for Bridgeland.

The problem: `signature` re-elaborates the declaration head out of context, but
many Bridgeland theorem heads rely on section parameters like `C`, `k`, `a`,
and `b`. Those are available in the original declaration context, but not in a
synthetic standalone `signature` block.

This produced errors like:

- `ComparatorManual.lean:279:8: Failed to elaborate signature`
- `ComparatorManual.lean:316:8: Failed to elaborate signature`
- `ComparatorManual.lean:454:8: Failed to elaborate signature`
- `ComparatorManual.lean:466:8: Failed to elaborate signature`
- `ComparatorManual.lean:692:8: Failed to elaborate signature`
- `ComparatorManual.lean:704:8: Failed to elaborate signature`
- `ComparatorManual.lean:765:8: Failed to elaborate signature`

This approach is no longer the active one.

### Attempt 8: revert to `prooflessModule`

This is the current state. `Site.lean` was patched back to `prooflessModule`,
`lake build exposition` succeeds in this repo again, and regenerated
`ComparatorManual.lean` now uses `prooflessModule` on the theorem-like TFB
entries instead of `signature`.

## What Has Been Found

### 1. The C-file rebuilds are real and expected under the current shadow policy

The shadow regeneration path still deletes the entire shadow directory:

- `writeComparatorShadow` currently does `IO.FS.removeDirAll shadowDir`
  around `Site.lean:1687-1689`.

That wipes:

- `bridgeland-shadow/.lake/packages`
- `bridgeland-shadow/.lake/build`

or any other cached local build state underneath the shadow.

Then `buildShadowSite` immediately does:

- `lake update`
- `lake exe cache get`
- `lake build comparatorManual`
- `lake exe comparatorManual`

Because `comparatorManual` is a `lean_exe`, Lake performs native compilation,
so you see large numbers of `.c` / `.o` builds. This is why the rebuild feels
so expensive.

This is the next structural problem that should be fixed.

### 2. `lake exe cache get` is not the full answer

Even when cache pull succeeds, it does not undo the cost of deleting the local
shadow build state. The user explicitly noticed this as "why are all the C
files getting rebuilt?"

### 3. Multiple concurrent Lake builds are harmful

At one point there were three simultaneous `lake build comparatorManual`
processes:

- one started directly by Codex,
- one started by `exposition --shadow-only`,
- one started manually by the user shell.

That makes the build slower and can create confusing filesystem churn.
The Codex-started duplicates were killed.

### 4. The `ELAN_HOME` error was likely not the core blocker

There was user-visible confusion around `ELAN_HOME`, but in the Codex shell the
commands were reproducible without that being the real root cause. Later the
user noted the terminal was not refreshing correctly when the shadow directory
was replaced underneath it. Treat the cold-shadow rewrite as the important
problem, not `ELAN_HOME`.

### 5. TFB ordering and filtering were improved

The TFB slice is no longer in the earlier rough order, and the selection is
less noisy than the initial extractDeps-style output. The dependency-layer view
is implemented in `Site.lean`.
